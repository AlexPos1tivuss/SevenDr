# ООО «Семь Драконов» — B2B Платформа оптовых продаж детских игрушек

Полнофункциональная веб-платформа для оптовой торговли детскими игрушками. Разработана для компании **ООО «Семь Драконов»** (г. Минск, Беларусь). Система охватывает полный цикл работы с клиентами: от регистрации и просмотра каталога до оформления заказа, отслеживания статуса и общения с администратором через встроенный чат.

## О проекте

«Семь Драконов» — B2B-платформа, ориентированная на оптовых покупателей: магазины, детские сады, развивающие центры и другие организации. Платформа предоставляет удобный каталог товаров с оптовыми ценами, систему управления заказами и инструменты для администраторов.

**Ключевые особенности:**
- Оптовая система ценообразования (скидки при покупке от 5, 20 и 50 единиц)
- Полноценная панель администратора для управления товарами, заказами и клиентами
- Встроенная система чата между клиентом и администратором по каждому заказу
- Загрузка логотипов компаний и изображений товаров
- Полностью русскоязычный интерфейс

## Функциональность

### Для клиентов (B2B-покупатели)

#### Регистрация и авторизация
- Регистрация с указанием реквизитов компании: название, УНП, ФИО директора, телефон, адрес
- Возможность загрузки логотипа компании при регистрации
- Вход по email и паролю
- Сохранение сессии в `localStorage`

#### Каталог товаров
- Просмотр полного каталога с карточками товаров
- Фильтрация по категориям, возрастным группам и материалам
- Поиск товаров по названию
- Отображение трёх уровней оптовых цен для каждого товара:
  - от 5 штук
  - от 20 штук
  - от 50 штук
- Индикатор наличия товара («В наличии» / «Нет в наличии»)
- Просмотр детальной информации о товаре в модальном окне

#### Корзина
- Добавление товаров в корзину с выбором количества
- Автоматический расчёт актуальной оптовой цены в зависимости от количества
- Редактирование количества товаров прямо в корзине
- Удаление товаров из корзины
- Счётчик товаров в шапке сайта в реальном времени
- Итоговая сумма с учётом оптовых скидок

#### Оформление заказа
- Заполнение адреса доставки
- Подтверждение и отправка заказа
- Автоматическое создание чата к каждому заказу

#### Мои заказы
- Список всех оформленных заказов
- Отслеживание статуса заказа в реальном времени
- Просмотр состава заказа и итоговой суммы
- Доступ к чату с администратором по каждому заказу

#### Чат с администратором
- Переписка в реальном времени с менеджером по конкретному заказу
- История сообщений
- Автоматическая прокрутка к последнему сообщению
- Отправка сообщений клавишей Enter

---

### Для администратора

Вход: `admin@admin.by` / `admin`

Панель администратора состоит из четырёх разделов:

#### 1. Заказы и статистика
- Общая статистика: количество заказов, выручка, количество клиентов, новые заказы
- Таблица всех заказов с возможностью изменения статуса
- Статусы заказа:
  - **Новый** — заказ только поступил
  - **В обработке** — менеджер приступил к работе
  - **Собирается** — товар комплектуется на складе
  - **Доставляется** — заказ передан в доставку
  - **Подтверждение** — ожидается подтверждение клиента
  - **Завершён** — заказ получен клиентом
- Возможность выставить дату доставки для каждого заказа

#### 2. Пользователи
- Список всех зарегистрированных клиентов
- Просмотр реквизитов компании (УНП, директор, телефон, адрес)
- Отображение логотипа компании клиента

#### 3. Чат с заказчиками
- Список всех активных чатов по заказам
- Двухпанельный интерфейс: список чатов слева, переписка справа
- Отправка сообщений клиентам
- Разграничение сообщений по отправителю (визуальное)
- Автоматическое обновление сообщений каждые 5 секунд
- Автопрокрутка к новым сообщениям

#### 4. Товары
- Таблица всех товаров в каталоге
- Добавление нового товара с полями:
  - Название, категория, описание
  - Возрастная группа, материал, страна-производитель
  - Три уровня оптовых цен
  - Загрузка изображения (файл с превью) или URL
  - Статус наличия
- Редактирование любого товара
- Удаление товара
- Превью загруженного изображения перед сохранением

### Принцип работы

1. **Frontend** — SPA-приложение на React. Маршрутизация реализована через Wouter. Все запросы к API проходят через TanStack Query, который кэширует данные и инвалидирует кэш после мутаций.

2. **Backend** — Express-сервер, обслуживающий как API (`/api/*`), так и статику фронтенда (`dist/public`). В dev-режиме Vite проксирует запросы. Все обращения к базе данных инкапсулированы в слой `storage`.

3. **Storage Layer** — единственная точка доступа к базе данных. Все роуты вызывают методы `storage.*`, никогда не взаимодействуя с Drizzle напрямую из роутов. Это упрощает тестирование и поддержку.

4. **Загрузка файлов** — Multer сохраняет загруженные изображения (логотипы компаний, фото товаров) в директорию `uploads/`. Файлы доступны по пути `/uploads/<имя_файла>`. При сохранении генерируется уникальное имя с расширением оригинального файла.

## API

Все эндпоинты используют префикс `/api/`. Формат ответа — JSON.

### Аутентификация

| Метод | Путь | Описание |
|---|---|---|
| `POST` | `/api/auth/register` | Регистрация нового пользователя (multipart/form-data, поддерживает загрузку логотипа) |
| `POST` | `/api/auth/login` | Вход по email и паролю |

### Товары

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/products` | Список всех товаров |
| `GET` | `/api/products/:id` | Один товар по ID |
| `POST` | `/api/products` | Создать товар (multipart/form-data) |
| `PUT` | `/api/products/:id` | Обновить товар (multipart/form-data) |
| `DELETE` | `/api/products/:id` | Удалить товар |

### Заказы

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/orders` | Все заказы (для админа) |
| `GET` | `/api/orders/user/:userId` | Заказы конкретного пользователя |
| `POST` | `/api/orders` | Создать заказ |
| `PATCH` | `/api/orders/:id/status` | Изменить статус заказа |

### Пользователи

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/users` | Список всех пользователей |

### Чаты и сообщения

| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/chats` | Все чаты (для админа, с данными о клиентах) |
| `GET` | `/api/chats/user/:userId` | Чаты конкретного пользователя |
| `GET` | `/api/chats/:id/messages` | Сообщения чата (отсортированы по времени) |
| `POST` | `/api/chats/:id/messages` | Отправить сообщение в чат |

### Статические файлы

| Путь | Описание |
|---|---|
| `/uploads/*` | Загруженные изображения товаров и логотипы компаний |

## Запуск проекта

### Требования
- Node.js 20+
- PostgreSQL (или доступ к Neon Serverless PostgreSQL)

### Установка зависимостей

```bash
npm install
```

### Настройка базы данных

Создайте базу данных PostgreSQL и пропишите переменные окружения (см. раздел ниже). После этого примените схему:

```bash
npm run db:push
```

### Режим разработки

```bash
npm run dev
```

Запускает сервер Express и Vite одновременно на порту **5000**. Vite работает в режиме HMR (горячая перезагрузка).

### Сборка для продакшна

```bash
npm run build
```

Собирает:
- Фронтенд: Vite → `dist/public/`
- Бэкенд: esbuild → `dist/index.js`

### Запуск в продакшне

```bash
npm run start
```

## Роли пользователей

### Клиент (обычный пользователь)
- Регистрируется самостоятельно
- Видит каталог, корзину, свои заказы и чат с администратором
- Не имеет доступа к панели администратора

### Администратор
- Флаг `is_admin = true` в базе данных
- Имеет доступ к полной панели администратора
- Может управлять товарами, менять статусы заказов, общаться с клиентами
- Не видит корзину (не предназначена для администраторов)

**Учётные данные администратора по умолчанию:**
```
Email:    admin@admin.by
Пароль:   admin
```

## Особенности реализации

- **Общие типы** — схема базы данных в `shared/schema.ts` используется и на фронтенде, и на бэкенде, что исключает дублирование типов
- **Оптовые цены** — цена в корзине пересчитывается автоматически при изменении количества товара
- **Корзина** — реализована через React Context (`CartProvider`), доступна во всём приложении без лишних запросов к серверу
- **Чат** — обновляется автоматически каждые 5 секунд через `refetchInterval` в TanStack Query
- **Загрузка изображений** — Multer сохраняет файлы с уникальным именем и оригинальным расширением; поддерживается превью перед отправкой
- **Валидация** — все входящие данные на бэкенде проверяются через Zod-схемы, сгенерированные из схемы Drizzle
